# 01. 엔터프라이즈 사내 지식 RAG

> 마지막 업데이트: 2026-06-24
> 수록 사례: 한국 17 · 글로벌 13 · 코드베이스 4 (총 34)

## 개요

엔터프라이즈 사내 지식 RAG는 2024~2026년 사이 가장 빠르게 production화된 LLM 활용 영역이다. 사내 위키·문서·티켓·코드·채팅 등 흩어진 지식을 단일 의미 검색 레이어 뒤에 모아 두고, 직원이 자연어로 질의하면 LLM이 출처를 인용해 답하는 패턴이 사실상 표준이 되었다. Microsoft 365 Copilot(Semantic Index + Graph), Glean(통합 인덱스 + 지식 그래프), Atlassian Rovo(BM25 + kNN + 기존 Jira/Confluence 그래프), Slack AI(연합 검색 + 권한 보존 RAG)가 플랫폼 진영을 이끌고, Klarna·Morgan Stanley·Cisco·Uber·LinkedIn·Spotify·DoorDash 같은 운영 기업은 자체 사내 봇으로 수십만 건의 티켓·문서를 처리한다.

한국에서는 네이버 플레이스(Backoffice AI Agent), 우아한형제들(물어보새·우아한공방), LY Corporation/LINE(FAA), 카카오엔터프라이즈(Koworker), SKT(A.X 4.0 기반 내부 RAG), 카카오뱅크(AI 플레이그라운드), 포스코홀딩스(Gemini 기반 지식 포털), 신한은행(AI ONE), 토스플레이스(PANDA)가 1차 자료를 공개했다. KB국민카드는 스켈터랩스 BELLA QNA로 RAG + 하이브리드 검색을 결합한 사례를 제시했다.

공통적으로 떠오르는 화두는 (1) 권한 인식 검색(permission-aware retrieval) — 사내 문서마다 ACL이 달라 같은 인덱스를 쓰면서도 사용자별 결과가 달라야 함, (2) 데이터 신선도와 청크 전략 — 정책·코드·티켓은 매일 바뀌므로 incremental indexing이 필수, (3) 평가의 어려움 — "80% 품질까지는 빨리, 95%까지는 대부분의 개발 시간"(LinkedIn) 패턴이 반복적으로 관찰되는 점이다. 단순 벡터 검색은 빠르게 한계를 드러내 하이브리드(BM25+벡터)·리랭커·지식 그래프·에이전틱 검색이 표준 스택으로 굳어지고 있다.

## 한국 사례

### 네이버 플레이스 — Backoffice AI Agent (RAG+MCP)
- **목적**: 플레이스 백오피스 운영팀의 내부 정책·매뉴얼·SQL·툴 지식 검색. 응답 품질과 속도 동시 향상.
- **데이터 소스**: 플레이스 백오피스 사내 문서, 운영 매뉴얼 등 팀 내부 문서.
- **스택**: LLM + RAG + MCP 기반 에이전트. 임베딩/벡터 검색에 추가로 MCP로 사내 툴 호출 연계.
- **아키텍처 특이점**: 단순 검색을 넘어 MCP로 백오피스 시스템 자체를 툴로 노출 → 에이전트가 "검색 + 실행" 모두 수행.
- **성과(공개 수치)**: 응답 만족도 ×2, 응답 속도 1.4×, 토큰 66% 감소, 툴 호출 49% 감소.
- **출처**: [네이버 플레이스 개발 블로그 — Backoffice AI Agent 구축기](https://medium.com/naver-place-dev/backoffice-ai-agent-%EA%B5%AC%EC%B6%95%EA%B8%B0-rag-mcp-%EA%B8%B0%EB%B0%98-%ED%94%8C%EB%A0%88%EC%9D%B4%EC%8A%A4ai-%ED%8A%B9%ED%99%94-%EC%A7%80%EC%8B%9D-%EA%B2%80%EC%83%89-%EC%8B%9C%EC%8A%A4%ED%85%9C-9a66b4afa1aa)

### 우아한형제들 — 물어보새 QAB (AI 데이터·지식 통합 에이전트)
- **목적**: 사내 구성원이 코드 없이 데이터 분석/사내 지식 검색. 데이터 전문가 24/7 self-service 대체.
- **데이터 소스**: 사내 데이터 카탈로그, 테이블·쿼리·로그 가이드, 정책 문서, 위키·Jira·Slack 통합. Slackbot UI.
- **스택(2024 v1)**: GPT-4o + LangChain + RAG + Text-To-SQL + LLMOps. 듀얼 파이프라인(RAG → 메타데이터/가이드 검색, Text-To-SQL → 쿼리 생성). 2023 우아톤 1위 → 2024-01 정식 TF.
- **스택(2026 v1.5)**: Amazon Bedrock + Claude. **슈퍼바이저 에이전트(하이브리드 에이전트)** 구조로 전환. 중앙 슈퍼바이저가 사용자 의도를 분류해 SQL에이전트(데이터 조회)·지식에이전트(문서 검색)·서포트에이전트(문의 자동화)로 작업을 라우팅. 4개 기능 — SQL 디스커버리(자연어 → SQL), 데이터 디스커버리(테이블·컬럼 추천), 지식 디스커버리(위키·Jira·Slack 통합검색), 서포트 자동화(반복 문의 AI 응답).
- **로드맵(2026 v2.0 예정)**: MCP(Model Context Protocol) 표준 도입. ReAct(추론·행동 반복) + Reflect(자가 검증) 구조의 멀티에이전트 시스템으로 진화.
- **성과**: 출시 1년 만에 전사 도입률 30% 이상. AWS re:Invent 2025 대표 사례로 발표됨.

### LY Corporation — PJ One Piece 분석 AI 에이전트 (Tech-Verse 2026)
- **회사/팀**: LY Corporation (LINE·Yahoo! JAPAN 통합 법인) 사업 부문 PM·PO 팀

## 코드베이스 RAG (별도 섹션)

### GitHub Copilot Enterprise
- **목적**: 조직의 사내 레포에 대한 코드 검색·설명·생성. 신규 개발자 온보딩.
- **데이터 소스**: GitHub 사내 레포, 이슈, PR, 위키.
- **스택**: 벡터 DB(임베딩) + GitHub의 비신경망 코드 검색을 결합한 하이브리드 RAG. Weaviate/Redis/PostgreSQL 등 stateful 서비스.
- **아키텍처 특이점**: 코드 변경 시 **임베딩 캐시 무효화**가 가장 흔한 production 인시던트 — 리팩토링 후 stale completion이 개발자 신뢰를 조용히 무너뜨림.
- **출처**: [GitHub Next — Copilot for Your Codebase](https://githubnext.com/projects/copilot-view/) · [Quastor — How GitHub Copilot Works](https://blog.quastor.org/p/github-copilot-works) · [Markaicode — GitHub Copilot RAG Architecture 2026](https://markaicode.com/architecture/github-copilot-rag-architecture/)

### Sourcegraph Cody Enterprise
- **목적**: 멀티레포 엔터프라이즈 코드 검색·생성. 최대 10개 레포 컨텍스트.
- **데이터 소스**: 사내 레포 + 코드 인텔리전스(LSIF/SCIP) 데이터.
- **스택**: 벡터 임베딩 + Sourcegraph Search API(BM25/심볼 검색) 하이브리드. Gemini 1.5 Flash로 최대 1M 토큰 컨텍스트. 코드 그래프(go-to-definition 등) 통합.
- **아키텍처 특이점**: **search-first RAG** — 벡터 단독 대신 코드 인텔리전스가 컨텍스트의 1차 출처. 엔터프라이즈 배포 시 RAG 전체가 고객 네트워크 안에서 실행 가능(코드 외부 유출 X).
- **출처**: [Sourcegraph Blog — How Cody Understands Your Codebase](https://sourcegraph.com/blog/how-cody-understands-your-codebase)

### Tabnine — Global RAG (멀티레포 + 에어갭)
- **목적**: 엔터프라이즈 다중 레포 코드 컨텍스트. 항공·금융 등 폐쇄망.
- **데이터 소스**: 워크스페이스 외부의 원격 레포 + 내부 코드.
- **스택**: Global RAG(중앙 색인) + RAG-based Connection. VPC/온프레/풀 에어갭 배포.
- **아키텍처 특이점**: IDE 워크스페이스에 열려있지 않은 레포까지 자동 컨텍스트. 다중 레포 환경에서 단일 레포 도구(Cursor 등) 대비 우위.
- **출처**: [Augment Code — Cursor vs Tabnine](https://www.augmentcode.com/tools/cursor-vs-tabnine)

### Cursor / Augment Code (엔터프라이즈 컨텍스트 비교)
- **목적**: 단일 워크스페이스 깊이(Cursor) vs 대규모 모노레포(Augment)에서의 코드 RAG.
- **스택**: Cursor — 로컬 임베딩 + IDE 통합. Augment Code — 엔터프라이즈 스케일 모노레포 인덱싱.
- **아키텍처 특이점**: Cursor는 모노레포에서 메모리 고갈/성능 병목이 알려진 한계. 엔터프라이즈는 보통 Sourcegraph/Tabnine/Augment 등 다중 레포 친화 도구로 보완.
- **출처**: [Augment Code — Sourcegraph Cody vs Cursor vs Augment](https://www.augmentcode.com/tools/sourcegraph-cody-vs-cursor-vs-augment-code-for-enterprise-development) · [Augment Code — Cursor freezes on large codebases](https://www.augmentcode.com/tools/why-cursor-freezes-on-large-codebases-5-alternatives)

## 이 도메인의 공통 패턴

- **하이브리드 검색이 사실상 표준**: BM25(키워드) + 임베딩(시맨틱) + 리랭커. Glean, Atlassian Rovo, KB국민카드 BELLA QNA, Sourcegraph Cody, Anthropic Contextual Retrieval 등 거의 모든 1차 자료가 채택.
- **권한 인식(permission-aware) 검색이 차별화 포인트**: Slack/Microsoft Graph/Cisco는 retrieval-time에 원본 시스템 ACL을 그대로 적용. 그렇지 않으면 ConfusedPilot류 권한 우회 리스크 발생.
- **지식 그래프로의 진화**: Glean(엔티티-관계 그래프), Atlassian Rovo(Jira/Confluence 기존 그래프), LinkedIn(이슈 KG로 -29% 해결시간), Microsoft Graph. 평면 청크의 한계가 production에서 드러남.
- **에이전틱 RAG (pre/post-retrieval agents)**: Uber EAg-RAG의 Query Optimizer + Source Identifier, 네이버 플레이스의 MCP 통합, Spotify AiKA의 MCP 확장. 단일 retrieve → generate를 넘어선 멀티스텝 패턴이 prod에서 정확도를 끌어올린다.
- **자체 임베딩 모델 fine-tuning**: 고객사/조직별로 임베딩을 fine-tune (Glean 월 1회, JPMorgan 다세대 진화). 클릭 데이터·피드백 루프로 self-improving.
- **Slack/Teams가 1순위 진입 UI**: Salesforce Techforce, Uber Genie, 우아한형제들 물어보새, Spotify AiKA, Atlassian Rovo, Glean. 별도 웹 UI보다 채팅이 사용량이 훨씬 높다.
- **온프레/에어갭 + 멀티 LLM 게이트웨이**: Cisco(LLM-as-a-Service), Tabnine(에어갭), Sourcegraph Cody(고객 네트워크 내 RAG), JPMorgan, 우아한형제들 Bedrock — 모델 락인 회피와 보안이 같이 간다.
- **자동화율보다 자동화한 부분의 품질**: Klarna 교훈 — 50% × 98% 정확도 > 75% × 90% 정확도. 위험도별 라우팅·저신뢰 회피가 필수.

## 자주 등장하는 실패 모드

- **임베딩 캐시 무효화 누락 → stale completion**: GitHub Copilot Enterprise 류 코드 RAG에서 가장 흔한 production 인시던트. 리팩토링·정책 변경 시 인덱스 갱신 지연이 신뢰를 조용히 침식.
- **권한 우회(ConfusedPilot 패턴)**: RAG-LLM이 임베딩에 들어간 비공개 문서 내용을 권한 없는 사용자에게 leak. M365 Copilot, 모든 Graph 기반 시스템에서 학계 경고.
- **마지막 15% 품질 long tail**: LinkedIn 보고 — 80% 품질은 빨리 도달, 95%까지가 대부분의 엔지니어링 시간. 평가셋·human-in-loop 부재 시 prod에서 침몰.
- **모호 쿼리·복합 쿼리 분해 부재**: Uber 사례에서 EAg-RAG 도입 전 정확도가 낮았던 주된 원인. Query Optimizer/Source Identifier 같은 pre-step이 필수.
- **단일 벡터 검색에 의존**: 키워드성 사내 식별자(사번, 시스템명, 약어)에서 벡터만으로는 부족. 하이브리드 검색 미도입 시 빠르게 천장.
- **데이터 신선도(stale docs)**: 정책/매뉴얼/티켓이 자주 바뀜에도 야간 배치로만 인덱싱 → 사용자가 옛 답변 신뢰. LY/FAA가 파인튜닝 대신 RAG를 택한 핵심 이유.
- **자동화 비율 과적합**: Klarna 사례 — 위험도 분류와 회피 응답 없이 자동화율을 끌어올리면 규제·민원 비용이 폭발.
- **청크 전략 빈약**: 카카오엔터프라이즈가 if(kakaoAI)2024에서 "청크가 핵심"이라 한 이유. 문맥 무관 토큰 분할 → retrieval miss.
- **평가 부재**: AI evals(factuality/relevance/hallucination)과 human-in-loop 없이 prod로 가면 회귀 탐지가 안 됨 (Morgan Stanley + OpenAI 협업의 출발점).
