# 02. 프로덕션 RAG 아키텍처와 스택 트렌드

> 조사 시점: 2026-06-10 · 범위: 2024 하반기 ~ 2026년 중반 1차 자료 위주

## 개요

2024년 하반기부터 RAG 스택은 "벡터DB 하나 + 임베딩 하나 + LLM 하나"라는 초기 형태에서 벗어났다. 프로덕션 표준은 **하이브리드 검색(BM25 + dense) + 리랭커 + 컨텍스트 보존 청킹 + 평가/관측 파이프라인**으로 굳어졌고, Anthropic의 Contextual Retrieval(2024-09) 공개가 이 흐름을 가속화했다. Anthropic은 Contextual Embeddings + Contextual BM25로 top-20 retrieval failure rate를 5.7%→2.9%(-49%), 리랭킹까지 결합 시 5.7%→1.9%(-67%)까지 떨어진다고 보고했다.

벡터DB 선택은 양극화되었다. 한쪽은 "기존 RDB에 얹는" pgvector/pgvectorscale, 반대쪽은 OpenSearch·Elastic·Vespa 같은 풀텍스트+벡터 통합 엔진이다. 순수 벡터DB(Pinecone, Qdrant, Weaviate, Milvus)는 5M~수십억 벡터 규모, 메타데이터 필터링 복잡도, 운영 부담에 따라 선택된다. 클라우드 매니지드 RAG(AWS Bedrock Knowledge Bases, Azure AI Search, Vertex AI Search, Snowflake Cortex Search, Databricks Vector Search)는 PoC→프로덕션 사이의 시간을 압축하는 기본기로 자리잡았다.

비용·지연 측면에서는 프롬프트 캐싱(Anthropic 최대 -90% 비용/-85% 지연, OpenAI 자동 -50%)과 임베딩 캐싱이 사실상 필수가 됐다. 평가는 RAGAS(개발) + LangSmith/Arize Phoenix/TruLens(운영) 조합으로 분화되었다. 멀티모달은 ColPali·ColQwen2 같은 vision-language late interaction이 PDF/표/차트를 OCR 없이 처리하는 길을 열었고, GraphRAG는 LazyGraphRAG(2024-11)로 인덱싱 비용을 1/1000 수준으로 떨어뜨려 실용성을 확보했다.

## 벡터DB 트렌드

### pgvector의 부상 — "5M 벡터 이하면 그냥 Postgres"
- pgvector는 **기존 Postgres 인프라가 있는 팀에 가장 실용적**. 별도 운영 부담 없이 ACL/트랜잭션/JOIN을 그대로 활용 가능.
- 한계는 스케일. 5M 벡터에서 pgvector p95 지연이 80~140ms(ef_search 설정 의존), 같은 조건의 Pinecone pod 기반 배포는 30ms 이하. 대부분의 "벡터DB 교체" 사례가 이 지점에서 발생.
- **pgvectorscale**(Timescale)의 등장으로 격차가 좁아져, Qdrant 등 전용 DB와 대등하게 경쟁한다는 벤치마크가 다수 나옴.

### 하이브리드 DB(OpenSearch / Elasticsearch / Vespa)의 강세
- 2024년 OpenSearch 2.19+ / Elasticsearch 8.14(retriever API, 8.16 GA) / Vespa는 **RRF(Reciprocal Rank Fusion)를 네이티브로 지원**. BM25 + dense를 같은 엔진에서 한 쿼리에 처리 가능.
- 이미 ELK·검색 인프라가 있는 조직은 "벡터DB 추가" 대신 "기존 엔진의 벡터 모드 켜기"로 갈아탄다. AWS Bedrock Knowledge Bases도 OpenSearch Service 매니지드 클러스터를 vector store로 정식 지원.

### 전용 벡터DB의 분화
- **Pinecone Serverless**(2024-01): storage/compute 분리, blob storage 위에 vector clustering. 10~100배 비용 절감 주장. 운영 부담 0, 수십억 벡터 스케일이 강점이나 락인과 비용이 트레이드오프.
- **Qdrant**: Rust 기반, 필터링이 강함. 메타데이터 복잡한 워크로드, 실시간성 요구에 강세. BM42(2024)로 sparse+dense를 단일 엔진에서 처리.
- **Weaviate**: GraphQL + 하이브리드 검색이 1급 기능. 모듈러 임베딩 파이프라인.
- **Milvus / Vespa**: 10억 벡터 이상 분산 운영 시 사실상 두 후보. Milvus는 compute/storage 완전 분리, GPU 가속. Vespa는 Yahoo 검색 출신으로 랭킹 표현력이 가장 풍부.
- **Chroma**: 로컬·프로토타이핑 단계의 디폴트로 자리잡음. Chroma Research가 임베딩·청킹 벤치마크를 활발히 공개.

### 클라우드 매니지드 RAG의 부상
- **AWS Bedrock Knowledge Bases**: chunking(fixed/hierarchical/smart) + embedding(Titan/Cohere) + vector store(OpenSearch Serverless, Aurora pgvector, Pinecone, Redis, MongoDB) + Bedrock 모델까지 풀스택 매니지드. Terraform 배포 패턴이 표준화됨.
- **Azure AI Search**: 하이브리드(BM25 + vector RRF) + semantic ranker(transformer cross-encoder) + agentic retrieval(2025 preview)을 단일 SaaS로 묶음. semantic ranker는 최대 50 결과까지 재정렬.
- **Vertex AI Search / RAG Engine**: GA 발표. Google 검색 품질을 엔터프라이즈에 이식하는 것이 포지션. RAG Engine은 vector DB·임베딩·LLM을 자유롭게 끼우는 어댑터 모델.
- **Snowflake Cortex Search**: 데이터가 이미 Snowflake에 있으면 별도 벡터DB 없이 hybrid search 서비스를 생성. TREC 규모 데이터셋에서 "1시간 내 프로덕션 서비스 생성"을 주장.
- **Databricks Vector Search**: Mosaic AI 평가/모델 서빙과 통합. 데이터 레이크하우스에 이미 있는 데이터에 대한 RAG가 주 시나리오.

## 임베딩 모델

### 상용 API
- **OpenAI text-embedding-3-large**(3072d, MTEB ~64.6): 디폴트 안전 카드. text-embedding-3-small이 비용/지연 최적화 대안.
- **Cohere embed v4**(MTEB ~65.2~66.3): search_document/search_query 입력 타입 분리, 멀티모달 지원. 다국어·엔터프라이즈에서 강세.
- **Voyage AI voyage-3 / voyage-3-large**(2024-09, 2025-01): voyage-3가 OpenAI v3-large 대비 평균 7.55% 우위(8개 도메인), 1024d로 vectorDB 비용 3~4배 절감. **voyage-finance-2, voyage-code-3, voyage-law-2, voyage-multimodal-3**처럼 도메인 특화 시리즈가 강점 — 금융·법률·코드 검색에서 4~16% 우위.

### 오픈소스
- **BGE 시리즈(BAAI)**: BGE-M3가 다국어·다단계(dense/sparse/colbert) 검색의 표준. BGE-reranker-v2-m3는 100+ 언어를 Apache 2.0으로.
- **E5 / multilingual-E5-large**: 한국어 포함 다국어에서 가성비 디폴트.
- **Nomic Embed**(2024-02): 완전 오픈(weights/data/code) 8K 컨텍스트, ada-002와 text-embedding-3-small을 단·장컨텍스트 모두에서 능가.
- **Stella-EN-1.5B-v5**: 영문 retrieval 벤치에서 OpenAI text-embedding-3-large를 전 지표에서 상회한 오픈모델.
- **GTE / mxbai / Jina v3**: 각각 고유 도메인·라이선스 강점.

### 트렌드 요약
- 도메인 특화(코드·법률·금융·의료) 임베딩이 범용 모델을 4~16%p 이상 상회 — Voyage가 이 시장을 선점.
- 차원 축소가 비용 절감 레버. 1024d / Matryoshka 임베딩이 표준화 중.
- 다국어/한국어는 BGE-M3 + multilingual-E5 + 한국어 특화 모델(KURE 등) 조합이 일반적.

## 청킹 전략

### Anthropic Contextual Retrieval (2024-09)
- **핵심 아이디어**: 각 청크 앞에 "이 청크가 전체 문서의 어디에 위치하는지 설명하는 50~100 토큰 컨텍스트"를 LLM으로 생성·prepend 후 임베딩.
- **결과**: Contextual Embeddings + Contextual BM25 → top-20 failure rate 5.7%→2.9%(-49%). 리랭커 추가 시 1.9%(-67%).
- 청크당 컨텍스트 생성 비용은 prompt caching으로 90% 절감 가능. 이 캐싱 가능성이 기법의 실용성을 뒷받침.

### Late Chunking (Jina AI, EMNLP 2024)
- arXiv 2409.04701. 전체 문서를 long-context 임베딩 모델(jina-embeddings-v2 8K, v3)에 통째로 넣고 **토큰 임베딩 시퀀스를 먼저 만든 뒤 mean pooling으로 청크 임베딩**을 생성.
- 청크 사이의 cross-reference(대명사·참조)가 보존됨. Naive 청킹 대비 cosine 유사도와 검색 성능이 일관되게 우위. Elasticsearch에 jina v2와 함께 통합되어 운영 사례가 누적 중.

### 의미 청킹 vs 재귀 청킹 — 실무 균형
- Chroma Research 등 벤치마크에서 **LLMSemanticChunker recall 0.919, ClusterSemanticChunker 0.913**. 그러나 RecursiveCharacterTextSplitter 400 tokens도 88~89%.
- 비용 대비 효과로 **재귀 512 tokens + 15% overlap + 메타데이터 강화가 디폴트**, 의미 청킹은 도메인 따라 ±9% 변동.
- 컨텍스트 클리프(약 2,500 tokens 이상에서 답변 품질 저하)로 인해 **실무 동작 범위 512~1024 tokens**가 합리적.

### Layout-aware / Page-aware
- PDF의 시각 구조를 무시하면 표·다단·헤더가 깨짐. Unstructured는 의미 요소 라벨링(title/list/table/figure)으로 청킹 로직을 구동.
- LlamaParse, Reducto, Docling, Marker-PDF가 OCR + 레이아웃 + 표 추출을 통합. **Docling/Marker는 self-host 오픈소스 디폴트**, **Reducto는 규제 도메인(금융·법률) 정확도 우위**, **LlamaParse는 LlamaIndex 통합 디폴트**.

## 검색·리랭킹

### 하이브리드 검색이 디폴트
- BM25는 **식별자·약어·희귀 용어·콜드스타트** 코퍼스에서 강함. Dense는 **paraphrase·동의어·교차언어**에 강함. 두 신호는 보완 관계.
- **RRF(score = 1/(k + rank), k=60)** 가 표준. 스코어 스케일 정규화 없이 합칠 수 있다는 점이 결정타. OpenSearch 2.12+, Elasticsearch 8.14+, Weaviate, Qdrant, Vespa, Azure AI Search가 모두 네이티브 지원.

### Sparse·Dense의 진화
- **SPLADE / SPLATE**: 학습된 sparse 표현. SPLATE는 ColBERTv2 임베딩에 sparse adapter를 붙여 PLAID와 동등 효과를 50개 후보 재랭킹 10ms 이하로 달성.
- **BM42**(Qdrant, 2024): BM25의 IDF에 transformer self-attention 가중을 결합. neural sparse의 실용 베이스라인을 노림.

### ColBERT / ColPali — 멀티벡터(late interaction)
- ColBERTv2의 late interaction은 token-level MaxSim으로 fine-grained 매칭. 한 문서를 토큰 임베딩의 bag으로 저장하는 비용 문제로 도입이 늦었으나, PLAID·ColBERT-live 등 인덱싱 최적화로 프로덕션화.
- **ColPali**(arXiv 2407.01449, ICLR 2025): PaliGemma + ColBERT 스타일 출력. **PDF 페이지를 이미지로 통째 임베딩**, OCR/레이아웃 파이프라인 제거. ViDoRe 벤치마크에서 SOTA. **ColQwen2**가 Llama 3.2 Vision 등과 결합되어 표·차트·다국어 문서로 확장 중.
- 실무에서는 **1차 검색 dense + BM25 → 리랭커**가 여전히 디폴트, ColPali는 **시각적 문서가 핵심인 도메인**(연구 논문, 재무 보고서, 발표 자료)에서 선택.

### 리랭커 사용 패턴
- **Cohere Rerank 3.5 / 4 / 4 Pro / 4 Fast**: API 기반 디폴트. v4 Pro는 품질, v4 Fast는 지연 최적.
- **BGE-reranker-v2-m3 (BAAI)**: 100+ 언어, Apache 2.0. GPU 호스팅 시 80ms 수준으로 Cohere와 경쟁. CPU에선 350ms로 한계.
- **mxbai-rerank, Voyage rerank, Jina reranker**: 도메인·언어·라이선스에 따라 선택.
- **사내 cross-encoder**: 도메인 데이터로 fine-tune하면 범용 reranker를 상회. 평가셋이 있어야 의미가 있음.
- 패턴: top-150 후보 → reranker top-20 → LLM 컨텍스트. Anthropic Contextual Retrieval 가이드의 디폴트도 동일.

## 질의 변환·확장

- **HyDE**: 질의 → LLM이 가상 답변 생성 → 그 답변을 임베딩해 검색. 의미 정렬 향상, recall은 별도. 짧고 추상적인 질의에서 효과적.
- **Multi-query**: 같은 의도의 여러 paraphrase 생성 후 각각 검색 → 결과 union 또는 RRF 결합. 표현 다양성이 큰 사용자 질의에 효과.
- **Step-back prompting**: 구체 질의 → 상위 개념 질의를 함께 생성. 추론이 필요한 multi-hop QA에 사용.
- **Query decomposition**: 복합 질의를 sub-question으로 분해 후 각각 검색·답변·통합. Haystack·LangChain·LlamaIndex가 기본 컴포넌트로 제공.
- 2025년 실무 흐름: **질의 변환을 항상 켜지 않는다**. 짧고 명확한 질의는 그대로 검색, 모호/복합 질의에 한해 라우터가 변환을 트리거(LLM 비용 절약).

## 평가

### 단계별 도구 분화
- **개발·CI**: RAGAS(faithfulness, answer_relevancy, context_precision/recall), DeepEval. 오픈소스, 정적 데이터셋 평가.
- **운영·관측**: LangSmith, Arize Phoenix(OTel 기반, 7.8k+ stars), TruLens(Snowflake 백킹). 실제 트래픽 트레이스 + LLM-as-judge.
- **엔터프라이즈 플랫폼**: Maxim AI, Galileo, Patronus AI 등 신규 진입.

### Golden set 패턴
- 도메인 전문가가 200~500 Q&A를 만들어 회귀 테스트. 검색 단계(recall@k, MRR)와 생성 단계(faithfulness, citation correctness)를 분리 측정.
- **LLM-as-judge**의 reliability를 별도로 검증(human spot-check, judge model 교체 비교)하는 흐름이 2025년 정착.
- 평가 인프라가 없는 RAG는 사실상 "운영 안 되는 RAG"라는 인식이 굳어짐.

### 벤치마크
- **BEIR**: 일반 정보 검색 표준.
- **MTEB**: 임베딩 모델 멀티태스크 표준. Leaderboard가 모델 선택의 1차 신호.
- **MIRACL**: 다국어 검색.
- **ViDoRe**(ColPali 논문 동반): visual document retrieval.
- **MS MARCO, NQ, HotpotQA**: 학술 baseline.
- **RAG-bench, RGB, CRUD-RAG**: RAG end-to-end.

## 비용·지연 엔지니어링

### 프롬프트 캐싱이 비용 곡선을 바꿈
- **Anthropic**(2024-08 베타, 2024-12 GA): cache write 1.25×~2× input, cache read 0.1× input. 최대 -90% 비용/-85% 지연(긴 시스템 프롬프트·문서 컨텍스트).
- **OpenAI**: 자동 prompt caching, -50% 자동. 1024 토큰 이상 prefix가 캐시 키.
- **Google Gemini**: explicit context caching, 시간 기반 과금.
- 설계 원칙: **정적 컨텍스트 앞, 동적 질의 뒤**. 시스템 프롬프트·검색 결과 묶음이 캐시 친화적으로 배치되도록 템플릿 분리.

### 임베딩 캐시 / 응답 캐시
- 31%의 LLM 질의가 직전 질의와 의미적으로 유사하다는 측정치. semantic cache(예: Redis Vector, GPTCache)가 p95 지연 2.1s → 450ms로 떨어뜨린 사례.
- 10-layer 캐싱(query/retrieval/rerank/chunk/response/KV/embedding/document/...)을 모두 도입한 사례에서 비용 80% 절감 보고.

### 임베딩 인프라
- **CPU vs GPU**: BGE-reranker v2-m3는 CPU 350ms / GPU 80ms. 트래픽 수준에 따라 손익분기.
- **배치/비동기**: 인덱싱은 배치 임베딩 API(OpenAI Batch -50%, Cohere batch)로 비용 절감. 실시간 쿼리는 별도 엔드포인트.
- **Fireworks, Together, Anyscale, Modal**: 오픈소스 임베딩·리랭커 호스팅 시장의 주축. 토큰당 비용이 OpenAI/Cohere 대비 낮음.

## 데이터 파이프라인·신선도

- "60% of enterprise RAG projects fail not because of retrieval/hallucination, but data freshness" 식의 진단이 2025년 들어 빈도 높게 등장.
- **Incremental indexing 표준 패턴**:
  1. 원본 시스템의 CDC(Confluence/SharePoint/S3/Git/DB)로 변경 감지.
  2. 영향받은 청크만 재임베딩(문서 hash, chunk hash 비교).
  3. 벡터DB upsert + 삭제 전파(orphan chunk 청소).
  4. 새 인덱스에 대한 회귀 평가(golden set 재실행).
- **삭제 전파**가 가장 흔한 버그. 원본에서 지워진 문서가 검색 결과에 남아 hallucination의 근거로 인용되는 패턴.
- **Streaming RAG**: 배치 야간 재인덱싱 대신 변경 이벤트를 Kafka/Kinesis로 흘려 retrieval index를 실시간 갱신. Confluent/Fivetran/Redpanda가 RAG 어댑터 출시.
- **TTL과 freshness 메타데이터**: 청크에 `updated_at`, `expires_at` 메타를 붙여 검색 시 신선도 가중 또는 필터링.

## 멀티모달 / 표·PDF·이미지

### PDF 파서 시장의 분화
- **Unstructured**: 의미 요소 라벨링(title/list/table/figure)을 청킹 로직과 결합. 가장 generic, 다양한 포맷.
- **LlamaParse**: GenAI 기반 파싱. LlamaIndex 통합 디폴트. 표·도형이 많은 복잡 문서에 강세.
- **Reducto**: agentic OCR 보정 레이어. 30+ 파일 타입. 금융·법률 규제 워크로드 정확도 우위.
- **Docling**(IBM): 오픈소스, markdown 출력. self-host 시 1순위.
- **Marker-PDF**: 오픈소스, LaTeX·수식 강함.
- 공통 성공 인자: **OCR 품질, 표 추출 정확도**. 프로덕션 문서 대부분이 "깨끗한 디지털 PDF"가 아니라 스캔·복합 레이아웃.

### Vision RAG / ColPali 계열
- ColPali·ColQwen2가 **OCR 파이프라인 자체를 제거**. 페이지 이미지 → multi-vector embedding → late interaction 검색.
- 트레이드오프: 인덱스 크기 큼, 벡터DB 지원 필요(Vespa, Qdrant, Weaviate가 multi-vector 지원). LLM은 vision-capable 필요(Claude 3.5 Sonnet, GPT-4o, Gemini, Llama 3.2 Vision).
- 적용 도메인: 연구 논문, 재무 보고서, 슬라이드, 의료 영상 보고서 — **표·차트·도형이 정보의 핵심인 곳**.

## GraphRAG / 지식 그래프 결합

- **Microsoft GraphRAG**(2024 공개, 오픈소스): LLM이 엔티티·관계 추출 → community detection → community summaries → global/local query에서 사용. 전체 코퍼스에 대한 "전역 질문"(이 데이터셋의 주요 테마는?)에 강함.
- **LazyGraphRAG**(2024-11): 사전 community summary 생성을 생략하고 쿼리 시점에 lazy 계산. **인덱싱 비용 1/1000(예: $10,000 → $10)**, global query 비용 1/700. 5,590 AP 뉴스 + 100 질의 벤치에서 8개 경쟁 방법 대비 96/96 head-to-head 승.
- **Neo4j + LLM**: 자체 GraphRAG 가이드/패키지 출시. property graph 기반 entity-aware retrieval.
- 패턴: **벡터 검색 + 그래프 traversal 하이브리드**. 벡터로 진입 노드 찾고, 그래프로 관련 엔티티 확장 후 컨텍스트 조합.
- 한계: 엔티티 추출 품질이 출력 품질의 천장. 도메인 특화 NER/스키마 설계가 필수.

## 보안·거버넌스 (ACL, PII, 감사)

### ACL-aware retrieval
- 표준 패턴: **인덱싱 시 청크 메타에 ACL(group/permission) 부착 → 검색 시 사용자 토큰 기반 필터링**. 벡터DB의 metadata filter로 구현 가능하나, 그룹·역할 변경의 propagation이 핵심 운영 과제.
- Bedrock Knowledge Bases, Azure AI Search, Vertex AI Search는 ACL 동기화 커넥터를 정식 기능으로 보유.

### PII redaction
- 인덱싱 전 사전 처리(Presidio, AWS Comprehend, Azure AI Language)로 PII 마스킹.
- 다국어 PII는 여전히 난제 — Meta 2024-10 평가에서 비영어 정확도 76.4%.
- **Inference-time redaction**: 답변 생성 후 PII detector로 후처리. 검색은 원본을 보되 출력만 마스킹하는 패턴이 GDPR/HIPAA 워크로드에서 일반화.

### Audit logging
- "어떤 청크가 어떤 사용자의 어떤 답변에 사용됐는가"를 immutable log에 저장 — 컴플라이언스(SOC2, HIPAA, GDPR right-to-erasure) 대응.
- LangSmith, Arize Phoenix, Maxim AI 같은 LLM observability 도구가 trace를 OTel로 내보내며 SIEM과 결합.

### 외부 표준화 흐름
- NIST: RAG-specific privacy guidelines 발간 진행 중.
- IETF: 2024-09 'Privacy-Preserving Retrieval Protocols' 워킹 그룹 구성.
- 예측: 2026년까지 엔터프라이즈 RAG의 85%가 privacy-preserving 기법을 표준 탑재(2024년 32% → 2026년 85% 전망).

## 한국 환경 특이점

### 한국어 임베딩 모델
- **KURE**(고려대 nlpai-lab, 2024-12): 한국어 검색 특화 임베딩. 대규모 query-document 쌍 + hard negative mining. 다국어/상용 임베딩 대비 한국어 벤치에서 우위. 장문(long-context) 검색에서도 경쟁력.
- **KF-DeBERTa**(Kakaobank): 금융 도메인 한국어 특화. 금융 용어/숫자/표 처리에 강점.
- **KoSimCSE**: SimCSE 기반 한국어 문장 임베딩. 오랜 기간 베이스라인.
- **다국어 보강**: BGE-M3, multilingual-E5-large가 여전히 광범위하게 사용. 영어 학습 비중이 높은 임베딩의 한국어 품질 저하가 일관되게 보고됨.

### 한국 벤치마크
- **Allganize RAG-Evaluation-Dataset-KO**: 도메인별 한국어 RAG 평가셋. 한국 RAG 평가 사실상 표준.
- **KorQuAD**: 한국어 독해 벤치(고전이지만 여전히 사용).
- **KomuRet**(2025 한글 및 한국어 정보처리 학술대회): 나무위키 450k 문서 + 1,454 질의 + 50k 후보. 구어·약어·오탈자·장문 구조 반영.
- **Open Korean Retrieval Benchmark** / ssisOneTeam Korean-Embedding-Model-Performance-Benchmark: 커뮤니티 주도 한국어 임베딩 리더보드.

### 한국 기업·솔루션
- **Upstage Solar**: SOLAR-mini가 AWS SageMaker JumpStart에 등재(2024-03). 한국어 특화 SLM으로 RAG 백엔드에 자주 채용.
- **Allganize**: RAG·LLM 파인튜닝 전문. Upstage 컨소시엄과 협업 발표. RAG 평가셋 공개로 한국 RAG 평가 표준화에 기여.
- **네이버 OCR**: 한국어 PDF 파싱에서 자주 등장하는 선택지.
- **규제**: 금융권은 망분리·온프레미스 요구로 **자체 호스팅 임베딩(BGE-M3, KURE) + 자체 호스팅 LLM(Solar, HyperCLOVA X) + 자체 벡터DB(Milvus/Qdrant on-prem)** 조합이 디폴트.

## 이 도메인의 공통 패턴

1. **하이브리드 + 리랭커 + Contextual chunking이 새 디폴트**. 단일 dense 검색은 PoC용.
2. **평가 인프라가 없는 RAG는 운영되지 않는다**. RAGAS + LangSmith/Phoenix 조합이 표준.
3. **PDF 파싱 단계가 RAG 품질의 천장**. 청킹·임베딩·리랭커를 아무리 튜닝해도 파서가 표를 깨면 끝.
4. **클라우드 매니지드 vs 자체 스택의 갈림길**이 명확해짐. 규제·온프렘은 자체 스택, 속도가 중요한 PoC/사내툴은 Bedrock/Azure/Vertex/Snowflake/Databricks.
5. **비용 곡선은 캐싱이 좌우**. 프롬프트 캐싱 + 임베딩 캐시 + semantic response cache 3중 구조.
6. **데이터 freshness가 가장 빈번한 실패 원인**. CDC 기반 incremental indexing은 옵션이 아니라 필수.
7. **다국어/한국어는 별도 트랙**. 영어 중심 임베딩의 한국어 품질 저하가 일관되게 보고됨 — KURE/BGE-M3/multilingual-E5의 도메인별 평가가 필수.

## 자주 보이는 실패 모드

- **고정 토큰 청킹 + 단일 dense 임베딩으로 시작 → 검색 품질 한계 → 임베딩 모델만 바꿔 보다가 실패**. 청킹/하이브리드/리랭커 셋이 함께 와야 함.
- **PDF 파서를 PyPDF/PDFMiner로 시작해 표·다단·헤더가 깨진 채 인덱싱**. 도메인 문서 종류 따라 Unstructured/LlamaParse/Reducto/Docling 비교 평가가 선행되어야 함.
- **삭제 전파 누락** — 원본에서 지워진 문서가 인덱스에 남아 검색되고, LLM이 confidently 인용. 가장 위험한 freshness 버그.
- **ACL 인덱싱 시점/쿼리 시점 불일치** — 어제 권한이 변경됐는데 벡터DB의 acl 메타가 어제 시점이라 정보 유출.
- **평가셋 부재** — "잘 되는 것 같다"로 출시 후 도메인 전문가 첫 테스트에서 hallucination 적발. golden Q&A 200~500개를 도메인 전문가가 미리 만들어야 함.
- **임베딩 모델/차원 교체 시 전수 재인덱싱 비용 누락** — 인덱싱 비용이 운영 비용의 90%인 경우가 흔함. Matryoshka 임베딩, 도메인별 동시 인덱스 운영으로 완화.
- **LLM-as-judge를 검증 없이 의존** — judge 모델 자체의 편향. 인간 spot-check + 다중 judge 비교 필요.
- **GraphRAG를 풀 인덱싱으로 시작** — $10k+ 비용 충격. LazyGraphRAG 또는 query-time graph construction 우선 고려.
- **프롬프트 캐싱 미적용** — 동일 시스템 프롬프트가 매 요청마다 fresh 토큰으로 청구. 트래픽 큰 워크로드에서 비용의 50~90%가 회수 가능한 누수.
- **한국어를 영어 모델로 그대로** — 도메인 정확도가 다국어/한국어 특화 모델 대비 일관되게 낮음에도 측정 없이 운영.
